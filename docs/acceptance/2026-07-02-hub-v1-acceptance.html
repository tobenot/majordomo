验收指南：中枢 Hub v1

确认 Bifrost 上报 → 中枢三张表 → 面板/复命 整条链路跑通 · 预计 10~15 分钟

0先搞懂这次验什么

上一次（bifrost-report-acceptance）只验了 Bifrost 一侧——中枢没建，所以「上报」= 落进离线缓存。这次中枢建好了，要验的是完整闭环：

窗口干活 → Bifrost report.ps1 → POST http://127.0.0.1:4350/ingest → 中枢更新三张表 → WebSocket 推面板 + persona 逐窗口复命。

三件事先记住：

  • 端口是 4350，不是 4317（4317 被企业微信占了）。daemon 的 WebSocket 和 HTTP /ingest 共用这一个端口。
  • 数据单向：v1 是只读仪表盘，面板不能对窗口发指令（那是以后的事）。面板能做的只有：勾/删待办、标记待验收已处理。
  • 离线缓存会自动补送：如果你之前跑过 Bifrost 但没开中枢，bifrost/cache/ingest.offline.jsonl 里的积压会在中枢第一次应答时被 report.ps1 顺带排空——所以你可能一开中枢就看到一堆历史窗口涌进来，这是对的。
  • 离线事件只录表、不扰人：积压回放的事件 ts 距今 > 5 分钟时，中枢只更新三张表，不触发 persona 复命、不推 Bark——离线的事你有记录就够了，不用指挥官处理。只有实时事件才会复命/推送。

1先自检：不开任何窗口，30 秒确认中枢能收

最快的信心来源——不用真窗口，直接手动打一条上报进去看反应。

① 起中枢（Web 面板会连它）

cd D:/GitRep/majordomo
npm run build          # 确保是最新代码
node dist/cli.js web   # 内嵌 daemon + Web 面板

日志里应出现 core daemon 监听 http+ws://127.0.0.1:4350（上报入口 /ingest）。浏览器会自动可访问 http://127.0.0.1:4351（web 端口 = daemon+1）。

② 另开一个 git bash，手动灌一条 Stop

curl -s -X POST http://127.0.0.1:4350/ingest \
  -H "Content-Type: application/json" \
  -d '{"windowId":"TEST-1","event":"stop","cwd":"D:/GitRep/demo","payload":{"text":"我把登录模块重构完了，抽出了 auth helper"}}'

预期返回 {"ok":true}。

③ 看面板

左栏应立刻冒出一个窗口 demo，状态「空闲」；点它，中间会显示一条 persona 人话复命（猫娘口吻，除非你配了真 API），下面是活动流一行 stop。

没有 persona API key 也没关系——离线模板会兜底出一句话。要验真人设，配 .env 的 PERSONA_API_* 再重开。

2灌满三张表：逐事件确认

继续用 curl 把六类事件都打一遍（改 windowId 可造多窗口）。对照下表看面板反应：

eventcurl payload 关键字段面板预期
session_startsource:"startup"左栏新窗口，状态 空闲
task_createdtaskId+taskSubject右栏「全局待办」+1（来源标窗口名），窗口转 干活中
task_completed同一个 taskId那条待办被划掉（确定性勾销，不烧 LLM）
stoptext:"..." + ts窗口转空闲；新鲜事件（ts 距今 ≤ 5min）触发 persona 复命；陈旧事件只录表不扰人
notificationnotificationType:"permission_prompt"窗口转 等你；右栏「待验收」+1；新鲜事件推 Bark，陈旧只录表
session_endreason:"prompt_input_exit"窗口状态 离线，圆点变灰

一条 notification 的例子：

curl -s -X POST http://127.0.0.1:4350/ingest -H "Content-Type: application/json" \
  -d '{"windowId":"TEST-1","event":"notification","cwd":"D:/GitRep/demo","payload":{"text":"要删 config.old，等你许可","notificationType":"permission_prompt"}}'

④ 验陈旧事件不扰人

发一条 ts 距今 > 5 分钟的 stop：

STALE_TS=$(node -e "console.log(Date.now() - 10*60*1000)")
curl -s -X POST http://127.0.0.1:4350/ingest \
  -H "Content-Type: application/json" \
  -d "{\"windowId\":\"TEST-STALE\",\"event\":\"stop\",\"ts\":$STALE_TS,\"payload\":{\"text\":\"10分钟前的老任务\"}}"

预期：左栏出现窗口、活动流有记录，但 没有 persona 复命、没有 Bark 推送。再发一条新鲜 stop（不带 ts 或 ts=现在）→ 立刻复命。这就是"离线回来只录表不炸手机"。

面板手动操作也验一下

  • 右栏待办框敲一条回车 → 冒出一条 manual 待办。
  • 勾选一条待办 → 划掉；点 × → 删除。
  • 待验收点「已处理」→ 该条变灰。
这些操作会广播给所有连着的前端（多开几个浏览器 tab 能看到同步）。刷新页面后状态还在（三张表落盘到 ~/.majordomo/hub-*.json）。

3真窗口端到端（已验收 ✅）

自检过了就可以上真的。中枢保持开着，另开一个新终端挂 Bifrost 干活：

cd D:/GitRep/majordomo
claude --plugin-dir ./bifrost

然后正常用：问一句（触发 stop）、让它列个三步计划做（触发 task_created/completed）、让它做需要许可的操作（触发 notification）、/exit（触发 session_end）。

面板应实时反映这个真窗口的一举一动，同时你本机还会听到 Bifrost 的提示音/弹窗（那是本机层，和中枢是两层接力）。

2026-07-02 实测发现：离线缓存排空时每条 stop 都触发 persona，十几条一起炸手机。已修：事件 ts 距今 > 5 分钟就只录表、不触发 persona/Bark。见 §2 ④ 的验证方法。

4验收清单

  • npm run build 通过；node dist/cli.js doctor 显示 Hub ingest : http://127.0.0.1:4350/ingest
  • curl 一条 stop 返回 {"ok":true}，面板左栏冒出窗口
  • 点窗口能看到 persona 人话复命 + 活动流
  • task_created 进待办，task_completed（同 taskId）自动划掉
  • notification 让窗口转「等你」且进待验收
  • session_end 让窗口变灰离线
  • 陈旧事件（ts > 5min）只录表不触发 persona/Bark，新鲜事件正常复命
  • 面板加/勾/删待办、标记待验收，多 tab 同步、刷新后仍在
  • （可选）真窗口挂 Bifrost，面板实时跟随
  • （可选）配 Bark 后，notification 和复命推到手机

5踩坑对照

现象原因 / 处置
curl 连接被拒中枢没起，或端口不是 4350。看 config.jsonc 的 port 与 bifrost/report.config.jsonc 的 ingestUrl 是否都指 4350。
面板一片空但 curl 返回 ok面板连的是 daemon 的 WS（同端口）。确认浏览器开的是 web 端口（4351），且右上角显示已连接。
真窗口没反应Bifrost 只在窗口启动时加载——必须开全新窗口。旧窗口不生效。
persona 复命很久才出/没出同一窗口有 personaThrottleMs（默认 15s）节流，高频 Stop 会跳过。想每条都出，把配置调小。
离线缓存排空时 persona/Bark 炸了已修：ts 距今 > 5 分钟的事件只录表不扰人。如果你仍被炸，确认 daemon 已重启（跑的是最新 build）。
一开中枢涌进一堆旧窗口正常——是离线缓存补送。现在不会触发 persona/Bark 了（见上条）。不想要记录就清空 bifrost/cache/ingest.offline.jsonl。
