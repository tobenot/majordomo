# 直连 deepseek-v4-pro Prompt Cache 特性与异常追踪

> 技术记录，持续维护。核心结论基于 2026-07-04 对本地 Claude Code 会话日志的实测。
> 所有数字可用文末脚本从 `~/.claude/projects/**/*.jsonl` 复核。
>
> ⚠️ **本文经历两次自我纠正**（TTL 假说 → 证伪 → 并发挤占）。第五节完整保留证伪过程作为方法论教训，请勿只读早期草稿的结论。

---

## TL;DR（当前最佳结论）

- 直连 `deepseek-v4-pro` 的 prompt cache **不是按时间过期的**（TTL 假说已证伪）。纯空闲时间与 miss 率无关：空闲 30s+ 仍只有 ~3% miss。
- **主因是并发挤占**：缓存池容量有限、疑似 LRU 淘汰，多个会话（你自己并发多开 + 看不见的外部租户）共用。别的请求插进你两轮之间，就把你的前缀挤掉 → 下一轮 miss。**已用剂量反应坐实**（插入越多 miss 越高）。
- 2026-07-04 直连线 miss 率从基线 ~2% 跳到 **8.9%**，**63% 的额外浪费来自 terrarium**——因为它全自动多开会话，自己挤自己。
- **残余基线 ~7%**：严格独占时仍存在，并发解释不了，归因未知（疑外部租户）。
- **压成本 = 减少并发直连**（串行 / 错峰），而非压单轮耗时——后者基于已证伪的 TTL 假说，无效。

## 一、口径与数据来源

**数据源**：`~/.claude/projects/<project>/<sessionId>.jsonl`，每条 `type=assistant` 消息的 `message.usage`。

**关键坑**：deepseek 代理模型的 `cache_creation_input_tokens` **恒为 0**。miss 率**不能**用 `create/(create+read)`（恒为 0，错误）。正确口径：

```
miss% = input_tokens / (input_tokens + cache_read_input_tokens)
```

| 字段 | 含义 |
|------|------|
| `input_tokens` | 本轮未命中缓存、重新计费的 prompt tokens（= miss 量） |
| `cache_read_input_tokens` | 命中缓存的 prompt tokens（= hit 量） |
| `cache_creation_input_tokens` | 恒为 0，不可用 |

---

## 二、核心证据：并发挤占（已坐实）

样本：今天全项目直连 deepseek-v4-pro **5,072 轮**。

### 检验 1 —— 精确插入

看本会话「上一轮 → 这一轮」的时间区间内，是否有**其他会话**的请求插进来：

| 两轮之间 | 轮数 | miss 率 |
|------|------|------|
| 有其他会话插入 | 607 | **18.0%** |
| 无其他会话插入 | 4,444 | **9.8%** |

有人插入，miss 率近乎翻倍。

### 检验 2 —— 剂量反应（坐实因果的关键）

两轮之间插入的**他人请求数** vs miss 率，单调上升：

| 插入他人请求数 | 轮数 | miss 率 |
|------:|------:|------:|
| 0 | 4,444 | 9.8% |
| 1 | 243 | 11.9% |
| 2 | 125 | 11.2% |
| **3-4** | 98 | **25.5%** |
| **5+** | 141 | **29.1%** |

**插得越多、挤得越狠、miss 越高。** 这不是"忙时段碰巧高"的相关，是剂量-反应，指向真实因果：并发请求挤占共享缓存池。

### 检验 3 —— 独占基线

严格独占（本轮前后 ±60s 内无任何其他直连会话）时：

| | 轮数 | miss 率 |
|------|------|------|
| 严格独占 | 1,913 | **7.2%** |
| 非独占 | 3,138 | 12.9% |

**并发能解释从 7% 到 13%+ 的抬升，但独占仍有 ~7% 基线**——这部分并发解释不了。

**机制推断**：缓存池容量有限、按 LRU（或类似）淘汰，多会话共享。空闲时间不触发过期（检验见第五节），只有"期间有别的请求来抢池子"才把你挤出去。独占残余的 7% 疑为**看不见的外部租户**（同一供应商其他用户共用池），但日志无法证实，**归因未知**。

| 项目 | miss% | uncached(烧钱) | cached | turns | 会话数 |

| **terrarium** | **12.3%** | **33.0M** | 234.9M | 2,495 | 12 |

### 为什么是 terrarium

terrarium 今天**全自动多开会话**（用户全程未参与）。多个自动会话并发跑，彼此的请求互相插入 → 互相挤占缓存池。wayfarer 并发度较低，受挤占较轻。**这是并发假设的直接现实来源。**

### 成本

terrarium 全直连累计 uncached **55.8M**，其中高 miss 事件贡献 **36.8M（66%）**。若无挤占理论上可省这部分。

## 四、排除项：不是这些原因

| 假设 | 排除依据 |
|------|------|
| 上下文塞满 | 出问题时 ctx 才 16~20 万；健康会话堆到 27 万仍 0% miss |

| system-reminder / compact / 动态注入 | terrarium 会话内计数为 0，无 `.claude/` 目录、无 compact 边界 |
| **缓存按时间 TTL 过期** | **已证伪，见第五节**。纯空闲时间与 miss 无关 |
| 慢工具（cargo）拖长轮间隔 | 部分为真（cargo 确实慢），但慢本身不导致 miss；是并发插入才导致 |

## 五、方法论教训：两次自我纠正（务必读）

本文结论演化了三版，前两版错误，完整记录以防重蹈：

### 弯路 1：小样本 → 误得 "82% 死区"
最初用**单会话小样本**（每个 gap 格仅 ~11 个）按轮统计，得出"gap 15-30s 死区 82% miss"。放大到全样本（16,720 轮）并按**事件**（合并共享 usage 的成对轮次）统计后，降为 15~42%。**教训：小样本格子的高百分比是噪声。**

### 弯路 2：因果倒置 → 误建 "TTL ~10s" 假说
第二版把 gap 越长越 miss 解读成"缓存 ~10s 过期"。**决定性检验**——把 gap 拆成两段：

- **t_idle（纯空闲）** = 上一轮结束 → 工具结果返回（模型没干活，纯时间流逝）
- **t_model** = 工具结果 → 本轮生成完（含重算 prefill）

| 时长 | miss vs t_idle | miss vs t_model |
|------|------|------|
| 短 | ~3% | ~1% |
| 中(10-20s) | **仍 ~3%** | 13~14% |
| 长(30s+) | **仍 ~3%** | 回落 |

miss 轮与 hit 轮的 t_idle 中位数 **0.1s vs 0.0s，几乎无差**；差异全在 t_model（7.7s vs 2.4s）。

**结论：时间不是原因。** "gap 越长越 miss"是因果倒置——一旦某轮 miss，重算 16 万 token 使 prefill 变慢 → t_model 变长 → gap 被撑大。**是 miss 撑大了 gap，不是 gap 导致 miss。** t_model 相关性纯机械（miss→必重算→慢），且发生在本轮生成时，不可能反过来导致本轮 miss。

**教训：相关的时间量若发生在"果"之后，极可能是果不是因。拆解时间轴、找真正先于事件发生的自变量（这里是"两轮间的他人插入"），才是干净的因。**

## 六、缓解方向

1. **减少并发直连（最有效）**：全自动多开会话 = 自己挤自己缓存。terrarium 那种批量自动会话应**串行或错峰**跑，而非并行。
2. ~~压单轮耗时 / cargo 后台化~~ → **无效**，基于已证伪的 TTL 假说。
3. **供应商侧**：向 deepseek 直连供应商确认缓存池是否 per-tenant 隔离、容量多大、淘汰策略——独占残余 7% 是否即外部租户挤占。
4. tokenhub 代理线是另一独立问题（稳定 53% miss，属路由层缓存未透传），不在本文范围。
