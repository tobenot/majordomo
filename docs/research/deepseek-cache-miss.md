> 所有数字可用文末脚本从 `~/.claude/projects/**/*.jsonl` 复核。

## TL;DR（截至当前）

- 2026-07-04 直连线 miss 率从基线 ~2% 跳到 **11.4%**（全体轮次口径），异常真实存在。
- **异常是阶跃式的**：上午（11-14 点）几乎全 0% miss，**下午 14:45~15:15 突然阶跃到 10~26%，之后一直高**。
- **阶跃跟着 terrarium 重度上场走，不跟并发走**：上午 wayfarer 高并发跑（350 轮/15min）miss 仍 0%；下午 terrarium 独占也 8%、放量即 26%。
- **已证伪**：①缓存不是按秒级 TTL 过期（官方：几小时~几天）；②不是别的用户/并发挤占（官方：缓存按用户隔离；且前几天同样并发无害）；③不是内容被每轮变动污染（miss 残留缓存平滑单调增，排除污染）。
- **挖到底的现象机制**：这条自动会话的持久缓存**只固化了最前 ~5-8K**，其余十几万 token 靠**易失短时缓存**，后者频繁接不上 → 每次退回 ~5-8K 重算（锯齿）。
- **根因未坐实**：为什么这条会话短时缓存频繁失效（节点调度？服务端波动？），**本地日志答不了，需 DeepSeek 服务端信息**。
- 用户体感"这个对话当时超慢"——与 miss 重算吻合（每次重算十几万 token）。

## 一、口径与数据来源

**数据源**：`~/.claude/projects/<project>/<sessionId>.jsonl`，每条 `type=assistant` 的 `message.usage`。

**关键坑**：`cache_creation_input_tokens` **恒为 0**。miss 率必须用：

```
miss% = input_tokens / (input_tokens + cache_read_input_tokens)
```

| 字段 | 含义 |
|------|------|
| `input_tokens` | 本轮未命中缓存、重新计费（= miss 量） |
| `cache_read_input_tokens` | 命中缓存（= hit 量） |
| `cache_creation_input_tokens` | 恒为 0，不可用 |

## 四、挖到底的机制：持久缓存只固化了最前 ~5-8K，其余靠易失短时缓存

### 决定性证据：miss 轮残留缓存完美单调增长

对最炸的 `479291e5`（532 轮）逐一提取 **miss 轮的残留 `cache_read`**（miss 时还剩多少命中），随 turn 变化：

```
turn 35 : 5120        turn 261: 7040
turn 88 : 5504        turn 364: 7680
turn 156: 6016        turn 522: 8320
```

**严格单调递增，精确对齐 128 token 边界。** `e02df83e` 同形态（5120→8064）。

### 这个形状排除了"内容被打断"假说

若是 Stop hook 注入 / 时间戳 / 计数器把前缀污染，残留应停在那个**变动位置**且**位置跳动**。但实测残留是**平滑单调、量化台阶** → 是**缓存 prefix unit 边界本身随会话增长**的特征，不是被污染。**"前缀被每轮变动打断"假说——排除。**

### 与官方机制拼合后的结论

对照 [官方文档](https://api-docs.deepseek.com/guides/kv_cache)："在请求边界和**固定 token 间隔**建立 cache prefix unit"。拼起来：

- 这条会话**只在最前 ~5-8K 固化了一个持久 prefix unit**（它随会话缓慢长大）。
- 其余十几万 token 靠**易失的短时缓存**（同节点还热时命中）。
- 每次 miss = 短时缓存没接上，裸露到只剩那个 ~5-8K 持久单元 → 重算中间十几万 token（锯齿）。
- 命中轮 = 短时缓存接上了，全量命中。

即：**不是内容污染，是这条会话的缓存"没能固化"**——大部分前缀停在易失层，频繁失效。

### 诚实的边界：为什么这条会话短时缓存频繁接不上，本地数据答不了

残留曲线确定了 **miss 时退到哪里**（~5-8K 持久单元），但**没解释**为什么这条自动会话的短时缓存频繁失效、而上午的会话一路命中。可能的最终诱因（**已超出本地日志能力，需 DeepSeek 服务端信息**）：
- 后端节点调度：长自动会话被换到冷节点
- 下午 DeepSeek 服务端波动（呼应用户"当时超慢"的体感，但命中轮延迟正常这点又不完全支持）
- 这条会话某种请求特征触发了短时缓存不固化

**结论**：本地日志已挖到底。能确定的是"缓存未固化、退到 ~5-8K"这个**现象机制**；触发短时缓存失效的**根因仍未知**，需服务端配合。

## 六、方法论教训：三次自我纠正（务必读）

结论演化了四版，前三版错误：

### 弯路 1：小样本 → 误得 "82% 死区"
单会话小样本（每格 ~11 个）按轮统计得"gap 15-30s 死区 82% miss"。放大到全样本 + 事件级后降为 15~42%。**教训：小样本格子的高百分比是噪声。**

### 弯路 2：因果倒置 → 误建 "TTL ~10s"
把"gap 越长越 miss"当成缓存过期。**决定性检验**：把 gap 拆成 t_idle（等工具结果，纯空闲）+ t_model（含重算）。miss 只跟 t_model 相关（7.7s vs 2.4s），跟 t_idle 无关（0.1s vs 0.0s）。**是 miss→重算→撑大 gap，不是 gap→miss。教训：相关的时间量若发生在"果"之后，极可能是果不是因。**

### 弯路 3：以偏概全 → 误信 "并发挤占"
用 07-04 单日数据得"插入越多 miss 越高"的剂量反应，以为坐实并发挤占。**但把同样分析拉到前几天**，发现前几天并发插入完全无害（有插入/无插入 miss 率相同）。**教训：单日内部相关不等于机制；必须用历史对照检验"为什么偏偏今天"。** + 官方文档明确缓存按用户隔离，跨用户挤占不成立。

### 弯路 4：想当然 → 误猜 "内容被每轮变动污染"
阶跃锁定 terrarium 自动循环后，直接猜"Stop hook 注入/计数器每轮打断前缀"。**残留缓存曲线证伪**：若是污染，残留应停在变动位置且跳动；实测残留平滑单调增、对齐 128 token，是缓存单元增长的特征，非污染。**教训：定位到"哪个会话/时段"≠定位到"机制"；下结论前先看数据形状（单调 vs 跳动）能排除一整类假说。**

## 七、缓解方向（暂定）

1. **根因未坐实前，最务实的止损**：这种**长自动循环会话**（Stop hook / autonomous loop 连轴转数百轮）是重灾区——它们缓存固化差、一旦 miss 单轮重算十几万 token，又慢又贵。**能拆短就拆短**（分段做、及时结束会话重开），别让单条会话滚到 500+ 轮。
2. ~~减少并发 / 错峰~~ → 证据不支持（前几天并发无害）。
3. ~~压单轮耗时 / cargo 后台化~~ → 基于已证伪的 TTL 假说，无效。
4. 服务端未整体降级（命中轮延迟正常），**换供应商能否解决取决于对方缓存固化策略**，不确定。
5. tokenhub 代理线是另一独立问题（稳定 53% miss，路由层），不在本文范围。
