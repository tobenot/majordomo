> 第六节完整保留证伪过程作为方法论教训。**只有第二节"已证实/已证伪"里的条目是定论，其余是假说。**

## TL;DR（截至当前）

- 2026-07-04 直连线 miss 率从基线 ~2% 跳到 **11.4%**（全体轮次口径），异常真实存在。
- **异常是阶跃式的**：上午（11-14 点）几乎全 0% miss，**下午 14:45~15:15 突然阶跃到 10~26%，之后一直高**。不是全天均匀。
- **阶跃跟着 terrarium 重度上场走，不跟并发走**：上午 wayfarer 高并发跑（350 轮/15min）miss 仍 0%；下午 terrarium 独占也 8%、放量即 26%。
- **已证伪**：①缓存不是按时间 TTL 过期（官方：几小时~几天）；②不是别的用户/并发挤占（官方：缓存按用户隔离）。
- **当前主假说（未坐实）**：下午 terrarium 跑的是 **Stop hook 驱动的全自动炼丹长循环**，循环里上下文有东西**每轮/每隔几轮在变**，破坏了 DeepSeek 的**前缀完全匹配**，导致缓存反复失效。这解释了"为何前几天没事、今天有事"（前几天没跑这种循环）。**下一步正在挖具体是哪段前缀在变。**
- 用户主观体感：**这个对话当时超慢**——与 miss 重算吻合（每次 miss 重算十几万 token，那轮就卡）。

## 一、口径与数据来源

**数据源**：`~/.claude/projects/<project>/<sessionId>.jsonl`，每条 `type=assistant` 的 `message.usage`。

**关键坑**：`cache_creation_input_tokens` **恒为 0**。miss 率必须用：

```
miss% = input_tokens / (input_tokens + cache_read_input_tokens)
```

| 字段 | 含义 |
|------|------|
| `input_tokens` | 本轮未命中缓存、重新计费（= miss 量） |
| `cache_read_input_tokens` | 命中缓存（= hit 量） |
| `cache_creation_input_tokens` | 恒为 0，不可用 |

## 二、已证实 / 已证伪（唯一的定论区）

### ✅ 已证实（硬事实，数据直接支持）
1. **07-04 miss 率异常暴涨**：逐日基线 1-2.7%，07-04 达 11.4%。
2. **异常是阶跃式**：上午 ~0%，下午 14:45 起阶跃到两位数并持续（见第三节时间线）。
3. **异常跟着 terrarium 走**：miss 高的时段都是 terrarium 重度活跃；terrarium **独占**（无其他会话插入）时下午也有 8-10% miss。
4. **命中轮服务端延迟正常**：07-04 命中轮延迟中位 2.3s，与前几天（2.4-2.6s）无异 → **服务端没有整体降级**。慢是 miss 重算的症状，非病因。
5. **DeepSeek 官方缓存机制**（[官方文档](https://api-docs.deepseek.com/guides/kv_cache)）：按用户隔离、落硬盘、存活几小时~几天、命中靠**前缀完全匹配**、best-effort 不保证 100%。

1. **"缓存按秒级 TTL 过期"**——证伪。纯空闲时间（等工具结果的时间）与 miss 无关，空闲 30s+ 仍只 ~3% miss；官方也说存活几小时~几天。
2. **"并发/别的用户挤占缓存池"**——证伪。①官方缓存按用户隔离；②**前几天同样两个项目并发跑，miss 率却正常**（见下表关键反证）。

## 四、当前主假说：前缀被每轮变动打断（未坐实，正在挖）

DeepSeek 官方：命中靠**前缀完全匹配**，`A+B` 一旦变成 `A+C`，其后全部 miss。

下午 terrarium 的两条大会话（`479291e5` 15:17 起 532 轮、`e02df83e` 18:10 起 608 轮）是 **Stop hook 驱动的全自动炼丹长循环**（任务："在尸体基础设施之上加 affordance 消费者"），无人看管连轴转。

**假说**：这种自动循环里，上下文中段有东西**每轮/每隔几轮变化**，破坏前缀匹配 → 缓存反复失效、锯齿。候选变动源（待逐一验证）：
- Stop hook 每次注入的反馈文本 / 条件字符串
- 递增计数、时间戳、轮次编号
- TodoWrite / 任务状态块
- 自动循环特有的、上午普通会话没有的任何注入

**为何符合所有事实**：
- ✅ 解释阶跃（上午没跑这循环，下午跑了）
- ✅ 解释 terrarium 独占也 miss（与并发无关，是自身前缀问题）
- ✅ 符合官方"前缀完全匹配"机制
- ✅ 解释锯齿（前缀被打断→miss→下轮又对齐→命中）

**注意**：之前排查 `479291e5` 曾说"无 system-reminder/compact"，但**没检查 Stop hook 每轮注入的内容**——那正是自动循环特有的嫌疑对象。下一步就是 diff 相邻轮的 prompt 前缀，定位失效字节位置。

## 六、方法论教训：三次自我纠正（务必读）

结论演化了四版，前三版错误：

### 弯路 1：小样本 → 误得 "82% 死区"
单会话小样本（每格 ~11 个）按轮统计得"gap 15-30s 死区 82% miss"。放大到全样本 + 事件级后降为 15~42%。**教训：小样本格子的高百分比是噪声。**

### 弯路 2：因果倒置 → 误建 "TTL ~10s"
把"gap 越长越 miss"当成缓存过期。**决定性检验**：把 gap 拆成 t_idle（等工具结果，纯空闲）+ t_model（含重算）。miss 只跟 t_model 相关（7.7s vs 2.4s），跟 t_idle 无关（0.1s vs 0.0s）。**是 miss→重算→撑大 gap，不是 gap→miss。教训：相关的时间量若发生在"果"之后，极可能是果不是因。**

### 弯路 3：以偏概全 → 误信 "并发挤占"
用 07-04 单日数据得"插入越多 miss 越高"的剂量反应，以为坐实并发挤占。**但把同样分析拉到前几天**，发现前几天并发插入完全无害（有插入/无插入 miss 率相同）。**教训：单日内部相关不等于机制；必须用历史对照检验"为什么偏偏今天"。** + 官方文档明确缓存按用户隔离，跨用户挤占不成立。
