# 直连 deepseek-v4-pro Prompt Cache 特性与异常追踪

> 技术记录，持续维护。核心结论基于 2026-07-04 对本地 Claude Code 会话日志的实测。
> 所有数字可用文末脚本从 `~/.claude/projects/**/*.jsonl` 复核。

---

## TL;DR

- 直连 `deepseek-v4-pro` 的 prompt cache **TTL 约 10 秒量级**。轮间隔（gap）越过 ~8s 后 miss 率随 gap 单调上升，10~24s 区间事件级 miss 率 **15~42%**，>25s 后反而回落。
- **命中与否几乎只由轮间隔决定**，与工具类型、上下文大小无关。
- 慢工具（Rust `cargo build/test`、长脚本）把轮间隔顶进 TTL 死区 → 缓存前缀失效、整段重算，表现为**锯齿**。
- 2026-07-04 直连线 miss 率从基线 ~2% 跳到 8.9%，**63% 的额外浪费来自 terrarium（Rust 项目）**，就是这个机制。
- **压成本 = 压单轮耗时**：让 gap 尽量 <8s（慢命令后台跑 / 合并 / 异步化）。

---

## 一、口径与数据来源

**数据源**：`~/.claude/projects/<project>/<sessionId>.jsonl`，每条 `type=assistant` 消息的 `message.usage`。

**关键坑**：这些 deepseek 代理模型的 `cache_creation_input_tokens` **恒为 0**。因此 miss 率**不能**用 `create/(create+read)`（恒为 0，错误）。正确口径：

```
miss% = input_tokens / (input_tokens + cache_read_input_tokens)
```

| 字段 | 含义 |
|------|------|
| `input_tokens` | 本轮未命中缓存、重新计费的 prompt tokens（= miss 量） |
| `cache_read_input_tokens` | 命中缓存的 prompt tokens（= hit 量） |
| `cache_creation_input_tokens` | 恒为 0，不可用 |

**gap 定义**：本轮 assistant 消息 timestamp 距上一轮的秒数。

**统计单位——事件级 vs 轮级（重要方法论）**：
deepseek 的一次 API 调用常对应日志里**连续两条 assistant 消息**（一条纯 text + 一条 tool_use），二者共享同一份 usage 快照。因此一次缓存失效会被"轮级"统计计两次，且第二条的小 gap 会稀释相关性。**必须按事件（合并 cache_read 相同的连续轮）统计**，gap 取事件首轮（触发轮）的值。下文 TTL 曲线均为事件级。

> 走过的弯路：最初用单会话小样本（每格 ~11 个）按轮统计，得出 "15-30s 死区 82% miss"。放大到全样本（16,720 轮 / 2,296 事件）并改用事件级后修正为下方曲线——趋势一致（gap 越大越易 miss），但绝对数值是 15~42%，非 82%。记此以防再被小样本误导。

## 二、核心发现：TTL 曲线（事件级，terrarium 全直连会话合并）

样本：**2,296 事件**（其中 miss 事件 120）。触发轮 gap vs miss 事件率，1 秒分辨率：

| **11** | 43 | 10 | **23%** |
| **12** | 30 | 11 | **37%** |

| 26~30 | ~45 | ~1 | ~2% |
| 30-60 | 76 | 4 | 5% |
| 60+ | 92 | 6 | 7% |

**miss 事件 gap 中位数 12.8s；命中事件 gap 中位数 3.2s。**

读法：
1. **gap ≤5s：几乎不 miss（0~2%）** —— 缓存还热。
2. **gap 6~8s：miss 率抬头（3~9%）** —— 接近 TTL 边缘。
3. **gap 10~24s：TTL 死区，miss 率 15~42%** —— 缓存已过期但重建窗口内，前缀命中点退回极靠前处。
4. **gap ≥25s：回落到 <7%** —— 大概率落入另一层长效缓存或已彻底重建，反而稳定。

TTL 边界大致在 **8~10 秒**起效，死区宽度到 ~24s。

## 三、锯齿现象（机制直观证据）

会话 `479291e5`（terrarium）连续轮次，`cache_read` 在满量 ↔ ~7.5K 之间一高一低跳，且**每次 miss 的触发轮 gap 都在 ~18-20s**：

```
turn  gap(s)  cache_read   miss%
 342  2.10     164,864     0%
 343  1.04     164,864     0%
 344 18.98       7,552    95%   ← gap 19s，缓存失效，命中点退回 7.5K
 345  0.79       7,552    95%   ← 同一事件第二轮（共享快照）
 346  2.54     165,632     0%   ← 立即重建，又全命中
 ...
 363  0.78     168,576     0%
 364  4.37       7,680    95%
 365  1.47       7,680    95%
 366  2.07     169,088     0%
 ...
 384 18.81       7,680    96%   ← gap 19s，又失效
 385  1.57       7,680    96%
 386  5.48     172,416     0%
 ...
 403 19.82       7,808    96%   ← gap 20s，又失效
 404 14.74       7,808    96%
 405  3.06     176,256     1%
```

每次 miss 的规律：**上一轮到本轮 gap ≈ 10~20s → 命中量从 ~17 万暴跌到 ~7.5K → 把中间 ~16 万 token 当未缓存重算**，下一轮 gap 回到 <5s 便立刻重新命中全量。上下文一直在 16~20 万，从头到尾没到窗口上限——**与上下文大小无关**。

对照两条健康会话（wayfarer）：开头 turn 1-13 有几次预热翻转（正常），之后一路 0% 再不翻转，因为其工具（读写文档 / Grep）秒回，gap 基本 <5s，从不进死区。

## 四、排除项：不是这些原因

| 假设 | 排除依据 |
|------|------|
| 上下文塞满 | 出问题时 ctx 才 16~20 万；健康会话堆到 **27 万** 仍 0% miss |
| 工具类型 | Read/Bash/Edit/Grep 在 terrarium 都 miss、在 wayfarer 都 0%；无单一工具专门触发 |
| system-reminder / compact / 动态注入 | terrarium 会话里 grep 计数为 0，无 `.claude/` 目录、无 compact 边界 |
| 某轮内容周期性变动 | 无周期性变动内容；变量是 gap 时间，非内容 |

## 五、2026-07-04 异常事件本体

### 逐日趋势（直连 deepseek-v4-pro，全项目）

| 日期 | miss% | uncached(miss) | cached(hit) | turns |
|------|------|------|------|------|
| 2026-06-29 | 1.7% | 1.3M | 76.5M | 761 |
| 2026-06-30 | 1.6% | 9.5M | 594.9M | 5,672 |
| 2026-07-01 | 1.9% | 8.2M | 432.4M | 4,345 |
| 2026-07-02 | 2.0% | 19.8M | 980.7M | 9,639 |
| 2026-07-03 | 2.2% | 5.1M | 231.6M | 2,308 |
| **2026-07-04** | **8.9%** ⚠️ | **52.2M** | 531.7M | 5,352 |

基线 1.6~2.2%，07-04 跳到 **8.9%**（miss 量翻 5~10 倍）。

### 07-04 按项目拆分

| 项目 | miss% | uncached(烧钱) | cached | turns | 会话数 |
|------|------|------|------|------|------|
| **terrarium**（Rust） | **12.3%** | **33.0M** | 234.9M | 2,495 | 12 |
| wayfarer-horizons（文档） | 5.6% | 19.7M | 332.5M | 3,232 | 18 |

52.2M miss 中 **terrarium 占 33M（63%）**，虽然它 turn 更少、上下文更小。

### 成本定位（terrarium 全直连会话累计）

- 总 uncached：**55.8M** tokens；总 cached：1,773M
- 其中**高 miss 事件（>50%）贡献 36.8M uncached（占浪费的 66%）**
- 即：若这些落进 TTL 死区的轮次能命中，理论上可省 ≈ **36.8M input tokens**

### 为什么偏偏是 terrarium

terrarium 是 **Rust 项目**，会话中 `cargo` 出现在 **107 处**，`cargo build/test`、跑 CSV 炼丹这类命令动辄 10~20s，大量轮次的 gap 天然落进 8~24s 死区。wayfarer 是文档/脚本项目，工具秒回，gap 基本 <5s。**同一个缓存机制，慢项目中招、快项目免疫。**

## 六、缓解方向

1. **压单轮耗时**（最直接）：慢命令（`cargo build/test`、长脚本）放后台跑 / 合并批处理 / 异步化，让轮间隔尽量落在 <8s。
2. **避开死区**：若某步必然慢（大编译），要么想办法压到 <8s，要么接受它 >25s（死区外反而稳）——最坏是卡在 10~20s。
3. **供应商侧**：向 deepseek 直连供应商确认 prompt cache TTL 是否可加长（当前实测 ~10s，偏短）。
4. tokenhub 代理线是另一个独立问题（稳定 53% miss，属路由层缓存未透传），不在本文范围。

## 八、复现脚本

脚本在 `~/mj_cache_tmp/`。核心逻辑：遍历 jsonl → 取每条 assistant 的 `message.usage` → 按第一节口径算 miss%。

| 脚本 | 作用 |
|------|------|
| `week_analyze.js` | 按 日期 × 模型 聚合 miss 率（逐日趋势表） |
| `cache_analyze.js` | 按会话聚合（找问题会话） |
| `deep_analyze.js` | 按项目聚合 + 高 miss 单轮的前置工具/上下文 |
| `trace_conv.js` | 单会话逐轮 miss 曲线 + 时间跨度 |
| `event_level.js` | **事件级** gap vs miss 率（TTL 曲线，本文核心） |
| `ttl_pin.js` | 1 秒分辨率 gap 分桶 + 成本估算 |
| `seq479.js` | 单会话连续轮次原始快照（看锯齿） |

复核最小步骤：
```bash
# 1. 列出某日直连会话
find ~/.claude/projects -name "*.jsonl" -newermt "2026-07-04 00:00:00"
# 2. 跑事件级 TTL 曲线
node ~/mj_cache_tmp/event_level.js
```
